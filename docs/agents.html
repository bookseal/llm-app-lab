Module 5 · Architecture & Agents

모든 에이전트 시스템의 핵심은 ~20줄 agent loop 하나. 나머지(Claude Code, Cursor, 커스텀)는 그 위에 얹은 "두꺼운 하니스"입니다. 이 모듈은 그 패턴들의 서베이예요.

📊 원본 슬라이드: ksetp.netlify.app/agents · 칩(슬라이드 N)이 덱의 번호와 같습니다.

The agent loop — the engine 슬라이드 5–6

Claude에 messages+tools 전송 → tool call 디스패치 → 결과 append → end_turn까지 반복.

flowchart LR A["messages + tools
→ Claude"] --> B{"stop_reason?"} B -->|tool_use| C["도구 실행"] C --> D["결과를 messages에 append"] D --> A B -->|end_turn| E["완료"]

Module 3의 Level 3에서 본 그 루프예요. 이 ~20줄이 엔진이고, 모든 화려한 에이전트는 이 위에 UI·권한·도구를 덧댄 것입니다.

What is Claude Code, actually? 슬라이드 7

기본 루프 + UI 레이어 + 권한 게이트 + ~25개 내장 도구 + MCP 도구 레지스트리 + 멀티스코프 메모리.

🏗️ 비유: 엔진(agent loop)은 같고, Claude Code는 그 위에 계기판(UI), 안전벨트(권한 게이트), 공구함(25개 도구)을 붙인 완성차예요. 지금 당신이 타고 있는 그 차.

Subagents — recursion for agents 슬라이드 8

제한된 도구셋·전용 system prompt로 자식 Claude 세션을 띄움 → 컨텍스트 격리·전문화.

부모는 요약만 받습니다 → 병렬 처리 + 재귀적 분해. 큰 일을 작은 독립 작업으로 쪼개 여럿에게 동시에 맡기는 셈이에요(42의 fork/프로세스 분할과 비슷한 감각).

Memory — what persists, where 슬라이드 9

"무엇을 어디까지 기억하나"를 5단계 스코프로 나눕니다.

Memory Scopes (5)
  • conversation — 현재 세션
  • session — 로그인 지속
  • user — 영속 선호
  • project — 레포 컨벤션(CLAUDE.md)
  • world — 벡터 스토어·지식베이스(= Module 4 RAG)

2편에서 "API는 stateless"라 했죠 — 이 스코프들이 바로 그 stateless 위에 우리가 얹는 기억 장치들입니다.

Patterns: routing · caching · async · progress 슬라이드 10–13

  • Multi-Model Routing — 싼 모델(Haiku)을 분류기로 써서 적절히 비싼 모델로 라우팅 → 비용 한정
  • Prompt Caching — 정적 system prefix를 cacheable로 표시 → 5분 윈도우 내 반복 비용 ~90%↓
  • Long-Running Tasks — 백그라운드 잡·예약 wake-up·webhook으로 비동기. 일시정지 중 상태를 어디 둘지 결정
  • Progress Streaming — 각 단계서 사람이 읽을 이벤트 emit, 전송계층과 분리, 종료는 done/error

Computer use & Safety 슬라이드 14–16

화면이 곧 API. 스크린샷 + 마우스/키보드로 API 없는 자동화. 단, 권한·한도가 필수.

🛡️ Safety = 권한 게이트 + 하드 리밋. 읽기는 항상 허용, 파괴적 작업은 확인, 실행은 샌드박스. 그리고 스텝 예산·토큰 캡·시간 데드라인·scope 제한으로 폭주를 막습니다.

에이전트가 무한 루프로 비용 폭탄이 되는 걸 막는 1차 방어선은?

능력(자율성)과 안전은 트레이드오프예요. 모델을 믿기보다, 코드로 강제하는 한도가 폭주를 막습니다. 파괴적 작업엔 사람 확인(human-in-the-loop)도 함께.
