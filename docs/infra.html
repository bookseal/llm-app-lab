부록 · Infra — 이 사이트는 어떻게 인터넷에 떠 있나

A.1 · nginx는 아는데 k8s는 처음이라면

이 클러스터는 결국 "여러 nginx 가상호스트를 자동 관리하는 시스템"입니다. 익숙한 nginx 개념에 1:1로 대응시키면 단번에 잡혀요.
If you know nginx, k8s here is just "auto-managed nginx vhosts." 1:1 mapping below.

A.2 · 요청이 들어오는 길 — the edge

https://llm-app-lab.bit-habit.com 한 번 누르면 이렇게 흐릅니다. 별도 로드밸런서 없이 Traefik 파드가 호스트의 80/443을 직접 물고 있어요.

flowchart LR U["Browser"] -->|"DNS: *.bit-habit.com (wildcard)"| IP["OCI public IP\n158.180.71.122"] IP -->|":80"| R["Traefik :80\n308 redirect to https"] R --> T IP -->|":443"| T["Traefik :443\nTLS termination (wildcard cert)"] T -->|"match by Host header"| ING["Ingress rules\n(per subdomain)"] ING --> SVC["Service (ClusterIP)"] SVC --> POD["Pod (app container)"]

핵심: 와일드카드 DNS *.bit-habit.com이 전부 이 서버 IP를 가리키므로, 새 서브도메인은 DNS 작업 없이 Ingress 규칙 한 줄만 추가하면 됩니다.
Wildcard DNS means a new subdomain needs only one Ingress rule — no DNS change.

A.3 · 와일드카드 TLS — 인증서 하나로 전부

서브도메인마다 인증서를 따로 발급하지 않아요. *.bit-habit.com 와일드카드 인증서 하나를 cert-manager가 자동 발급·갱신하고, 모든 서브도메인이 그걸 공유합니다.

sequenceDiagram participant CM as cert-manager participant DNS as Route53 DNS participant LE as Let's Encrypt participant T as Traefik CM->>DNS: write _acme-challenge TXT (DNS-01) LE->>DNS: verify TXT LE-->>CM: wildcard cert (*.bit-habit.com) CM->>CM: store in Secret "tls-secret" T->>CM: serve cert by SNI Note over CM,T: auto-renew before the 90-day expiry
💡 이 사이트(llm-app-lab.bit-habit.com)는 인증서를 새로 발급하지 않았어요. 기존 와일드카드(tls-secret)에 그대로 "무임승차"합니다 — 그래서 배포가 즉시 HTTPS.

A.4 · 배포의 현실 — push 하면 어떻게 live가 되나

이 레포는 git push → ① GitHub Pages(정적) + ② GitHub Actions가 서버에 ssh로 git pull(이 사이트). 인프라 레포는 원래 ArgoCD 자동동기를 의도했지만 실제로는 ArgoCD 미설치 → 수동 kubectl apply로 돕니다(정직하게).

flowchart TB DEV["git push (main)"] --> GH["GitHub repo"] GH -->|"Actions: pages.yml"| PAGES["GitHub Pages\n(static mirror)"] GH -->|"Actions: deploy-server.yml\nssh + forced git pull"| SRV["server clone\n/home/ubuntu/workspace/llm-app-lab"] SRV -->|"hostPath mount"| POD["nginx Pod (k3s)"] POD --> URL["llm-app-lab.bit-habit.com"]
세 가지 "push → live" 방식 · the trade-off
  • cron pull — 폴링. 단순하지만 빌드·피드백 없음. 표준 아님.
  • GitHub Actions (push CI/CD) — 이 레포가 쓰는 방식. 정적 → Pages, 서버 → ssh pull.
  • ArgoCD (pull GitOps) — 클러스터가 git을 감시·자동동기. k8s 표준이나 여기선 아직 미설치.

🔒 CI가 서버에 ssh하는 키는 forced-command 제약이 걸려 있어 오직 그 폴더의 git pull만 실행할 수 있어요(셸·포트포워딩 차단). 키가 새어도 피해가 한정됩니다.

A.5 · 이 사이트 자체의 배포 정의

k8s 리소스 3개로 끝나요: Deployment(nginx가 docs/를 hostPath로 마운트) + Service + Ingress(와일드카드 인증서 재사용). 이미지 빌드도 없어요.

무엇이 무엇을 담당하나 · separation of concerns
  • 콘텐츠 = llm-app-lab 레포의 docs/ — 서버에 클론되어 hostPath로 마운트. 갱신 = git pull.
  • 배포 정의 = bit-habit-infra 레포의 apps/llm-app-lab/ — Deployment·Service·Ingress YAML.
  • 둘은 다른 레포: 앱은 "내가 뭔지"만, 인프라 레포는 "이 클러스터에서 어떻게 도는지"만 압니다.
📂 매니페스트: bit-habit-infra/apps/llm-app-lab/{deployment.yaml, ingress.yaml} · 콘텐츠 마운트 경로: /home/ubuntu/workspace/llm-app-lab/docs

A.6 · 미니 퀴즈

이 사이트(llm-app-lab.bit-habit.com)의 TLS 인증서는 어떻게 마련됐나요?

와일드카드 인증서 tls-secret가 이미 *.bit-habit.com 전부를 커버하므로, Ingress가 그 Secret을 참조하기만 하면 즉시 유효한 HTTPS가 됩니다. cert-manager 어노테이션을 일부러 빼서 새 발급 시도를 막고 무임승차해요.

README는 "ArgoCD가 자동 배포"라는데 실제 클러스터를 확인하니?

argocd 네임스페이스·CRD·파드가 전부 없었어요. 문서는 의도를 적어둔 것이고 현실은 수동 배포예요. 그래서 인프라 문서는 항상 클러스터에 직접 물어보며 읽어야 합니다.
